Auditable Workspaces for AI Agents: What h5i Records and Why

Git records what changed and stops there. An auditable workspace records the rest — the prompt, the model, the agent's reasoning, the commands, the tests, and the risk signals behind every change — versioned in your repo, no SaaS required.

Published 2026-06-03 Updated 2026-06-22 Reading time 9 min read Tags Version control · Provenance · AI agents
In one line: An auditable workspace is a Git-backed worktree where every prompt, model, decision, command, log, policy, and handoff that produced a code change is recorded in your repo and provable after the fact. h5i implements it as a Git sidecar — code stays in normal commits and branches, while the AI context lives in refs/h5i/* and travels with the repo.

Git records what changed — and stops there. It can't tell you who wrote a line (agent or human), what prompt produced it, what context the agent used, or whether the result was tested. When Claude Code, Codex, or another coding agent writes a patch, the diff is the smallest part of the reviewable artifact. The rest — the prompt that started the task, the files the agent inspected, the assumptions it made, the tests it ran, the uncertainty it flagged, the handoffs it exchanged — is exactly what review depends on, and exactly what plain Git throws away.

h5i records that rest. This post explains what an auditable workspace stores, why each layer matters, and how the pieces fit together — starting from the version-control angle most teams hit first. For the canonical definition and the full sandbox story, see the cornerstone post on auditable workspaces.

What does an auditable workspace record?

An auditable workspace stores code changes plus the agent context that produced them. Workspace = worktree + prompt + model + commands + logs + policy + messages + PR evidence. In a practical system, that means prompts, model and agent identity, reasoning milestones, tool events, test results, risk signals, and reviewer handoff data are stored in a durable, queryable form tied to commits and branches.

The key design constraint is compatibility. Developers should still be able to use git log, git diff, pull requests, branches, and normal CI. The AI layer rides alongside Git as a sidecar, it does not replace it, and it never forces every teammate into a new source-control platform.

Why isn't Git alone enough?

Git tracks snapshots and object history — and nothing about how they came to be. It does not know that a change was generated from a prompt. It does not know whether an agent read the migration file before editing the model. It does not know that the agent chose a smaller refactor because the larger one looked risky. Commit messages can describe some of this, but they are manually written summaries, not reliable evidence.

That gap matters during review. A human reviewer needs to know where to focus. A compliance reviewer needs to know whether AI participated. A maintainer debugging a regression needs to know which intent introduced the behavior. A second agent resuming the work needs the last agent's working memory, not only the final patch.

What the workspace records

A useful auditable workspace stores seven kinds of information:

  • Prompt provenance: the user request or task summary that caused the change.
  • Agent identity: the tool and model used, such as Claude Code, Codex, or another agent.
  • Context trace: observations, decisions, actions, notes, and milestones captured during work.
  • Evidence coverage: which files or facts the agent actually inspected before editing.
  • Review signals: uncertainty, blind edits, large churn, scope creep, and prompt-injection indicators.
  • Test evidence: commands run, pass/fail result, and relevant output summaries.
  • Collaboration history: handoffs, review requests, risks, acknowledgements, and done messages.

h5i implements those ideas as a Git sidecar. Provenance records live in h5i refs, context can be recalled with h5i recall context, commits can be captured with h5i capture commit, and multi-agent messages can be stored in refs/h5i/msg. Plain Git remains plain Git.

Does auditing AI code require another model?

A strong audit layer should avoid depending on another model to explain whether the first model behaved safely. Some checks can be deterministic: Was the edited file observed? Did the agent record uncertainty? Did a prompt-injection phrase appear in an observed page? Did tests run? Did the diff touch files outside the requested scope?

Models are useful for coding, summarizing, and suggesting fixes. Audit records should be durable even when models change, vendors change, or transcripts are unavailable. That is why Git-native (sidecar) storage is attractive: the data can be fetched, pushed, reviewed, backed up, and inspected with normal developer tooling.

Where h5i fits

h5i treats Git as the transport and durability layer. It records AI provenance with commits, stores versioned reasoning as context, creates review surfaces for pull requests, and lets agents exchange typed messages over a shareable Git ref.

The result is not a replacement for review. It is a better review packet. Instead of asking a reviewer to infer intent from a patch, h5i gives them the patch, the prompt, the agent's working context, and the risk signals that say which parts deserve attention first.

When do you need an auditable workspace?

You need an auditable workspace when AI-generated code becomes routine rather than experimental. One-off autocomplete can survive with normal Git hygiene. Agentic development needs stronger records because the agent can edit multiple files, run tools, hand off work, and make architectural choices across a long session.

The trigger is simple: if your team would ask, "why did the agent do this?" after a bad merge, you should capture the answer before the merge.

FAQ

What is an auditable workspace for AI agents?

An auditable workspace is a Git-backed worktree where every prompt, model, decision, command, log, policy, and handoff that produced a code change is recorded in your repo and provable after the fact. h5i implements it as a Git sidecar: code stays in normal commits and branches, while the AI context lives in refs/h5i/* so it travels with the repo, with no SaaS.

Is an auditable workspace different from Git?

It is an additional layer on top of Git, a Git sidecar. The best implementations keep commits and branches normal while storing AI-specific metadata in sidecar refs or other durable records, so git log, git diff, pull requests, and CI keep working unchanged.

Is a prompt enough provenance for AI-generated code?

No. The prompt explains the request, but not the evidence the agent used, the decisions it made, the tests it ran, or the risks it introduced. A useful workspace records prompt, model and agent identity, context trace, evidence coverage, review signals, test results, and collaboration history.

Can an auditable workspace work with both Claude Code and Codex?

Yes. The useful abstraction is not one vendor's transcript format. It is a stable record of agent, prompt, context, action, test, and review data. h5i integrates with Claude Code via hooks and MCP and with Codex via explicit prelude, sync, and finish commands.
