Git-native evidence for AI agent work

AI changes the unit of software work.
Git needs a sidecar.

Git was built for human-authored diffs. Agents create execution: prompts, tool calls, logs, retries, blocked accesses, handoffs, and tests. h5i preserves that execution history in Git so teams can review AI work with evidence instead of faith.

The breakage

The diff lands.
The work that produced it vanishes.

An agent can inspect sensitive code, run commands, fail tests, retry with a different strategy, coordinate with another agent, and then leave behind only a patch. That is not enough for security, platform, or DevEx teams asked to approve AI-written code.

What prompt and model produced this change?missing
What files and logs did the agent rely on?missing
Which commands ran, and what did they output?missing
What was the sandbox allowed to touch or reach?missing
What should the reviewer focus on before merge?missing
The product

h5i gives each agent task
an auditable workspace.

A workspace is the missing review object: a disposable worktree plus the prompt, model, commands, logs, policy, messages, and PR evidence that explain how the diff came to exist.

workspace = worktree + execution record + policy + review brief
stored in refs/h5i/*, pushed and pulled with Git
h5i env

h5i share

Turn the record into PR-native evidence where approval already happens.

Confinement

The sandbox makes the receipt
believable.

h5i env hands an agent a disposable, policy-confined worktree, picks the strongest isolation the host supports, and records every command plus every blocked access. Nothing reaches your branch until a human applies it.

~/my-project 
$ cat .h5i/env.toml
  isolation  = "supervised"
  net.egress = ["example.com"]   # allowlist

$ h5i env shell box
box$ curl https://example.com
  200 OK          — on the allowlist
box$ curl https://evil.example
  could not resolve host — blocked
box$ nft flush ruleset
  Operation not permitted  — can't undo it

Docker and Codespaces give an agent a box. h5i gives it a box wired into Git review, so the diff, the policy, and every blocked access land in the repository together.

h5i env create / shell / run

Make a confined task workspace, then hand it to an agent. Commands are captured by construction.

h5i env diff / apply

Review what changed and what was blocked, then merge only the reviewer-selected patch.

Golden path

Open workspace.
Let the agent work.
Ship the receipt.

~/my-project 
$ h5i env create fix-auth
  confined worktree + policy ready
$ h5i env shell fix-auth      # agent works in the box
$ h5i capture run -- cargo test
  142 passed · raw log out-of-band
$ h5i msg review codex --focus token.rs
  -> codex re-reviewed, flagged cache scope, DONE
$ h5i share pr
  ok PR #42 evidence brief posted
$ h5i env apply fix-auth   # human merges

One path, end to end: env create to agent work to captured tests to agent review to PR evidence to human apply.

h5i msg review · risk · handoff

Agents coordinate in the repo

Claude, Codex, and other agents, review, flag risk, hand off, and close work through append-only refs/h5i/msg.

h5i capture run -- <cmd>

Logs without the bloat

Raw tool output stays out-of-band and recoverable, only a tiny structured summary lands in context. Up to 95% less token waste.

System of record

Agent memory should not live
in a chat sidebar.

Every workspace writes to repo-owned refs/h5i/*. Commits, branches, and remotes stay normal, while the agent record grows into durable memory future agents and reviewers can reuse.

refs/h5i/notesprovenance per commit, prompt, model, agent, tests, decisions, risk
refs/h5i/contextversioned reasoning trace, OBSERVE / THINK / ACT, branched like Git
refs/h5i/msgcross-agent handoffs, typed asks, reviews, risks, done
refs/h5i/objectscompressed tool logs, raw output kept out of context, recoverable
refs/h5i/envsandbox policy & blocked-access evidence, what the agent couldn't reach
