Auditable workspaces for AI agents

The auditable agent workspace workflow

h5i gives Claude Code, Codex, and other coding agents a Git-backed workspace: sandboxed worktrees, prompt-aware commits, compressed logs, agent handoffs, and review-ready PR evidence. Git tracks the diff. h5i tracks the workspace behind it.

Updated 2026-06-19 Scope current source CLI Tool Git sidecar (Rust CLI + Git refs)
An auditable workspace is the place an AI agent does its work, a Git-backed worktree where every prompt, decision, command, log, policy, and handoff is recorded in your repo and provable after the fact. Workspace = worktree + prompt + model + commands + logs + policy + messages + PR evidence.

The whole record lives in your Git (refs/h5i/*) and travels with the repo, no SaaS, no lock-in, works offline. h5i is not a Git replacement, not a hosted sandbox; it's a Git sidecar for auditable agent workspaces.

The golden path

Every workflow below is the same journey: open a workspace, hand it to an agent, let evidence accrue (commands, logs, prompts, handoffs), let a reviewer read the provable record on the PR, then a human applies and merges. h5i env, the sandboxed worktree, is the canonical auditable workspace and the entry point.

golden-path
# 1. Open the auditable workspace (sandboxed worktree + policy)
$ h5i env create fix-auth

# 2. Agent works inside the box (or you do) — commands/logs captured
$ h5i env shell fix-auth
$ h5i capture run -- cargo test

# 3. Commit with provenance (prompt, model, agent, tests, audit)
$ git add src/auth.rs tests/auth.rs
$ h5i capture commit -m "add token refresh flow" --audit

# 4. Hand off for review
$ h5i msg review codex "token refresh ready" --focus src/auth.rs

# 5. Post review-ready PR evidence, then apply/merge
$ h5i share pr post --style review
$ h5i env apply fix-auth

The workspace stack

Each layer is a property of the same auditable workspace, provenance (who/why/what the agent knew), confinement (what it couldn't reach, provable), and governance (deterministic audit, no model in the loop).

stack
Agent Workspace
├─ Sandboxed worktree        h5i env
├─ Prompt-aware commits      h5i capture commit
├─ Compressed tool logs      h5i capture run
├─ Agent handoffs            h5i msg
├─ Risk/audit signals        h5i audit
└─ PR evidence brief         h5i share pr

Install and initialize

h5i stores sidecar data under Git-owned state and pushable refs/h5i/* refs that travel with the repo. Run h5i init once per repository.

install
$ curl -fsSL https://raw.githubusercontent.com/h5i-dev/h5i/main/install.sh | sh
# or build directly from source
$ cargo install --git https://github.com/h5i-dev/h5i h5i

$ cd your-project
$ h5i init

Flow A: Solo developer with Claude Code

A day in an auditable workspace: you open a confined worktree, Claude Code works inside it, evidence accrues automatically, and you review the provable record before merging. h5i hook setup wires Claude Code's hooks once; from then on file reads, edits, prompts, and stop-checkpoints flow into the context trace.

solo-claude
# once: wire Claude Code hooks + MCP tools
$ h5i hook setup

# open the workspace and hand the box to Claude Code
$ h5i env create ship-oauth
$ h5i env shell ship-oauth -- claude

# evidence accrues as the agent works (prompts, reads, edits, logs)
$ h5i capture run -- cargo test
$ git add src/auth.rs tests/auth.rs && h5i capture commit -m "OAuth token refresh"

# review the provable record, then merge
$ h5i env diff ship-oauth
$ h5i env apply ship-oauth

Flow B: Claude + Codex review loop

One agent proposes, another reviews, on the same Git-backed evidence. Claude works in its workspace and proposes; Codex pulls the shared refs/h5i/* evidence, reviews the provable record, and applies on its own clone. Handoffs are Git-backed (h5i msg, the "Agent Radio"), so nothing depends on a shared chat window.

claude-codex-loop
# Claude: work in the box, commit with provenance, then propose
$ h5i env create refactor-client
$ h5i env shell refactor-client -- claude
$ h5i env propose refactor-client
$ h5i msg review codex "client refactor ready" --focus src/api/client.rs
$ h5i share push

# Codex (other clone): pull the evidence, review, apply
$ h5i share pull
$ h5i env inspect refactor-client --capture latest
$ h5i msg ack 1 "reviewed, applying"
$ h5i env apply refactor-client

Flow C: Security-sensitive refactor

When the work touches auth, secrets, or anything you'd rather an agent didn't run loose in your tree, the workspace's confinement pillar carries the weight: "the agent physically could not exfiltrate" beats "we logged it." h5i env create picks the strongest isolation the host can enforce (Landlock + seccomp, and a network egress allowlist on the container tier). Every blocked access and every command is recorded; nothing reaches your branch until you apply.

secure-refactor
$ h5i env probe
# show what isolation the host can enforce (incl. rootless Podman)
$ h5i env create rotate-secrets --isolation container
# strongest isolation: confined fs + syscall deny-list + egress allowlist
$ h5i env shell rotate-secrets
# work inside the box — or hand it to an agent
$ h5i env log rotate-secrets
# audit: what changed, every command run, every blocked access
$ h5i audit review --limit 50
$ h5i env propose rotate-secrets
# mediated commit + review brief — apply refuses without it
$ h5i env apply rotate-secrets

Flow D: Team PR review

The reviewer never has to take the agent's word for it. Provenance, audit signals, and the captured command record travel with the repo as refs/h5i/* and surface as review-ready PR evidence, a sticky comment with review-focus files, goal, evidence, checklist, audit signals, and per-commit provenance. Teammates sync the sidecar with h5i share.

team-review
# author: publish the provable record onto the PR
$ h5i share pr post --style review
$ h5i share push

# reviewer: pull the evidence and read the record Git can't show
$ h5i share pull
$ h5i recall blame src/auth.rs --show-prompt
$ h5i audit review --limit 50
$ h5i recall context show --depth 2

Commit with provenance

The provenance pillar of the workspace. Use h5i capture commit when staged changes are ready: it records the prompt, model, agent, decisions, tests, causal links, and audit output alongside the Git commit. In Claude Code the human prompt is captured automatically by the UserPromptSubmit hook; --intent is the fallback for Codex, CI, or manual commits.

capture
$ git add src/auth.rs tests/auth.rs
$ h5i capture commit -m "add token refresh flow" \
  --agent codex \
  --model gpt-5 \
  --intent "finish OAuth token refresh with tests" \
  --tests \
  --audit

Recall what happened

Reviewers and follow-up agents use recall commands to recover the provable record Git cannot show by itself.

recall
$ h5i recall log --limit 10
$ h5i recall blame src/auth.rs --show-prompt
$ h5i recall context show --depth 2
$ h5i recall notes coverage --commit HEAD
$ h5i recall resume

Audit risk before merge

The governance pillar: deterministic audit and compliance, no model in the loop. audit commands split high-precision quality signals from informational shape signals. A large diff alone should not block review; a large diff with blind edits, secrets, test regression, or code execution deserves attention.

audit
$ h5i audit review --limit 50
$ h5i audit scan
$ h5i audit policy check
$ h5i audit compliance --since 2026-01-01 --format html --output audit.html

Publish review-ready PR evidence

The workspace's record is useful before, during, and after review. h5i share pr post turns it into review-ready PR evidence: a sticky comment containing review-focus files, goal, evidence, checklist, audit signals, THINK/NOTE highlights, and per-commit provenance.

share
$ h5i share pr post --style review
$ h5i share pr body --style review
$ h5i share pr post --style replay --dry-run

# Sync sidecar refs with teammates:
$ h5i share push
$ h5i share pull

PR evidence styles

Use --style review when reviewers need a triage-oriented view of the evidence. Use receipt for a screenshot-friendly summary, detective for a narrative brief, replay when the reasoning DAG should be the hero, and minimal for quiet internal provenance.

Rule of thumb: open the auditable workspace first (h5i env), capture commands and commits while the agent works, audit before merge, and post review-ready PR evidence through the style that matches the moment. It all lives in your Git (refs/h5i/*), no SaaS, no lock-in.
Try it on your next AI-assisted branch: create a sandboxed workspace, capture the run, and post a review-ready PR brief. h5i env create my-branch && h5i capture run -- cargo test && h5i share pr post --style review
